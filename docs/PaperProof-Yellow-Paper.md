# PaperProof 核心合约黄皮书

## 摘要

PaperProof 是一个面向数字工件发布、验证、互动与激励的 Sui 原生协议。它的核心目标不是把所有内容、所有业务逻辑、所有社区行为都塞进链上，而是在 Sui 与 Walrus 之间建立一套清晰的协议分工：原始内容存储于 Walrus；工件身份、版本、类型、所有权、治理状态、关键元数据与可验证事件锚定在 Sui；链下 indexer、前端、评分系统和空投系统围绕这些可信锚点重建更丰富的应用层语义。

PaperProof 的合约设计倾向于“小核心、高约束、可治理扩展”。协议内置少数高价值 artifact type，例如预印本、技术报告、数据集、软件发布等；新增高价值类型不通过任意用户自助注册，而是通过治理与合约升级逐步进入核心协议。这种路径牺牲了一部分无许可类型扩张速度，换来更强的类型语义、索引一致性、长期可维护性和社区治理质量。

PPRF 是连接治理、费用、生态参与和未来激励的协议资产。它在当前合约中承担治理权、持币证明和生态参与权重的基础角色，但协议刻意避免把所有强激励立即绑定到单一行为事件上。PaperProof 的事件系统为后续 indexer、评分、贡献统计和空投提供原始证据，但这些事件需要在官方对象绑定、行为质量和反作弊规则下被解释，而不是被机械地视为最终奖励依据。

## 1. 协议定位

PaperProof 处理的是“可验证数字工件”的生命周期，而不仅是文件上传或内容展示。

一个数字工件可以是一篇预印本、一份技术报告、一个数据集、一个软件发布包、一篇长文或一个通用文件。它的价值不只来自内容本身，也来自围绕它产生的可验证关系：

- 谁发布了它；
- 它属于哪一种高价值类型；
- 当前版本是什么；
- 历史版本如何演进；
- 原始内容如何在 Walrus 上定位和校验；
- 谁拥有该 series；
- 该 series 的评论空间和点赞账本是什么；
- 它是否经过协议认可的类型规则；
- 它产生了哪些链上事件；
- 未来 indexer 和社区如何围绕它计算声誉、贡献和激励。

因此，PaperProof 的核心合约不是一个单纯的“文件登记表”，而是一套面向工件、类型、治理、互动和生态记账的基础协议。

## 2. 与现有平台的关系

PaperProof 不是 Twitter、Telegram、GitHub、arXiv 或其他内容与社交平台的替代品。它不试图在实时讨论、社交传播、代码协作、学术审核、内容分发、用户关系链或社区运营体验上与这些平台正面竞争。

这些平台各自已经形成强大的应用层能力：

- Twitter 更擅长公开传播、叙事扩散和短内容讨论；
- Telegram 更擅长群组组织、即时沟通和社区运营；
- GitHub 更擅长代码托管、协作开发、issue、pull request 和软件工程流程；
- arXiv 更擅长预印本分发、学术领域归档和研究者习惯；
- 传统内容平台更擅长推荐、流量分发、创作者关系和用户体验。

PaperProof 的定位不在这些应用层能力上硬拼。它要解决的是另一层问题：数字工件如何获得可验证身份、可追踪版本、可治理类型、可组合互动、可验证事件和可长期解释的生态记账。

因此，PaperProof 更像是一个协议层补充，而不是一个平台层替代。一个预印本仍然可以在 arXiv 上分发，在 Twitter 上传播，在 Telegram 社区里讨论，在 GitHub 上维护相关代码；PaperProof 提供的是围绕这件工件的链上身份、版本承诺、Walrus 内容锚点、评论与点赞对象、治理约束和未来激励证据。

这种互补关系使 PaperProof 能服务更大的天地。它不要求用户放弃已有平台，也不要求内容只存在于某一个前端。相反，它让不同平台上的内容、讨论、代码、数据和社区行为有机会被同一个可验证协议身份连接起来。

PaperProof 的核心价值在于把“内容被发布在哪里”与“内容在协议上被如何承认”区分开。前者属于应用平台，后者属于协议基础设施。

## 3. 设计原则

### 3.1 原始内容不上链

PaperProof 不把 PDF、数据集、软件包等原始内容写入 Sui 对象。原始内容属于大体积、低频读取、高存储成本数据，应由 Walrus 这样的去中心化存储网络承载。

Sui 链上只保存足以验证和组织内容的高价值状态，例如内容哈希、Walrus 标识、内容类型、artifact code、版本 ID、所有权和关键元数据。

这种分工使协议既能保留公开验证能力，又避免链上对象膨胀成为系统长期负担。

### 3.2 强约束状态上链

凡是会影响协议身份、类型语义、版本关系、治理权限、费用路径、评论空间绑定或未来可信索引的状态，都应受到链上约束。

例如：

- artifact type 必须来自官方 registry；
- series 和 version 必须由官方发布路径创建；
- comments tree 与 likes book 必须由官方 publishing 流程绑定；
- series metadata 可以更新，但必须受 owner 权限和长度约束；
- version metadata 一经提交不可修改；
- 治理对象必须绑定官方 root、vault 和 config；
- 关键状态变更必须产生事件。

这些约束使 PaperProof 的链上状态可以成为 indexer、前端和后续生态系统的可信入口。

### 3.3 少数内置高价值类型

PaperProof 不把 artifact type 设计成任意字符串市场。协议内置少数高价值类型，并通过治理控制启停、激活、费用与未来扩展。

内置类型的意义在于提供稳定语义。`preprint` 不是一个普通标签，而是协议承认的一类发布对象；`software_release` 也不仅是用户自填分类，而是可围绕软件包哈希、源码哈希和版本信息建立索引规则的类型。

这种设计使后续前端、搜索、学术索引、内容推荐、社区评分和空投统计都可以依赖类型语义，而不是在无限自由标签中猜测用户意图。

### 3.4 治理驱动扩展

PaperProof 的类型空间和协议能力应逐步扩张。新增类型、启停类型、费用调整、治理参数调整和关键权限迁移都应通过治理或受控升级完成。

这不是为了把协议变慢，而是为了防止核心协议被低价值类型、垃圾类型或短期叙事污染。PaperProof 的核心层应该稀缺、清晰、可验证；更自由的表达可以留给 metadata、前端分类、链下索引和社区应用。

### 3.5 事件是证据，不是结论

PaperProof 通过事件向 indexer 和社区系统公开工件发布、版本追加、评论、点赞、治理、费用和 metadata 更新等行为。

但事件本身只是证据。一个 comment event 不自动等于高质量贡献；一个 like event 不自动等于可领取激励；一个 publish event 也不自动等于内容真实、原创或有学术价值。

链下系统应以官方对象 ID 为可信入口，过滤非官方对象噪声，并结合行为质量、时间窗口、身份、内容验证、反滥用规则和社区治理解释事件。

## 4. 系统分层

PaperProof 可以理解为四个互相衔接的层次。

### 4.1 内容层

内容层由 Walrus 承载。它保存实际文件、PDF、数据包、软件包或其他 blob。PaperProof 只要求链上记录足以定位和校验内容的信息，例如 blob id、object id、digest、content hash 和 content type。

内容层关注“数据在哪里”和“数据是否匹配承诺”。

### 4.2 状态层

状态层由 Sui Move 合约承载。它记录 artifact series、artifact version、artifact type、官方 root、registry、fee manager、governance vault、governance config、comments tree 和 likes book 等协议对象。

状态层关注“协议承认什么”。

### 4.3 事件层

事件层连接链上状态和链下系统。发布、追加版本、更新 metadata、评论、点赞、治理结算、权限变化等关键操作都会产生事件。

事件层关注“发生了什么”。

### 4.4 应用层

应用层包括前端、indexer、搜索、推荐、评分、空投、分析、社区治理面板和第三方工具。

应用层关注“如何解释和使用这些事实”。

PaperProof 的核心合约不试图替应用层做所有判断。它提供可信事实、强约束边界和可组合对象，让应用层在此基础上发展。

## 5. 核心对象模型

### 5.1 PaperProofRoot

PaperProofRoot 是协议的根锚点。它记录官方 TypeRegistry、FeeManager、GovernanceVault 等关键对象 ID，并承载若干官方创建能力。

对前端和 indexer 来说，PaperProofRoot 是判断“这是不是官方 PaperProof 状态”的最高可信入口。任何声称属于 PaperProof 的 registry、vault、fee manager、comments tree 或 publishing event，都应最终能回到官方 root 绑定。

### 5.2 TypeRegistry

TypeRegistry 管理 artifact type 的协议状态。它决定哪些类型是协议承认的、哪些类型处于启用或停用状态、哪些类型可以继续发布或追加版本。

TypeRegistry 的存在使 PaperProof 的类型系统不是任意文本，而是可治理、可审计、可升级的协议层资源。

### 5.3 ArtifactSeries

ArtifactSeries 表示一个工件的长期身份。它不是某一个文件版本，而是围绕同一个内容对象或项目形成的连续发布单元。

Series 包含类型、owner、artifact code、当前版本、版本列表、comments tree、likes book、metadata extensions 等状态。它是前端展示、indexer 聚合、社区互动和未来激励统计的重要中心对象。

Series owner 可以在协议允许范围内更新 series metadata，并可以转移所有权。所有权变化会影响对应 comments tree 的管理边界。

### 5.4 ArtifactVersion

ArtifactVersion 表示某个 series 下的一次具体发布。它记录版本级内容承诺，例如内容哈希、Walrus 信息、内容类型和版本 metadata。

Version 的核心语义是不可变性。一个 version 一旦发布，不应再被修改成另一个内容对象。后续修订应通过追加新 version 表达，而不是覆盖旧 version。

这种模型使 PaperProof 既支持内容演进，又保留历史可验证性。

### 5.5 CommentsTree

CommentsTree 是 series 的评论空间。它支持评论发布、回复关系和状态标记。

评论状态包括 active、hidden、deleted 等语义。它们不是内容真实性判断，而是围绕展示、治理和社区管理建立的状态层。tree owner 可以隐藏或管理评论；comment author 的权限受到状态机限制，不能覆盖 tree owner 的管理决定。

CommentsTree 与 LikesBook 已经解耦。评论写入和点赞写入不再争用同一个互动对象，这更符合热门内容的并发需求。

### 5.6 LikesBook

LikesBook 记录某个 artifact series 的点赞状态。当前点赞语义是 PPRF 持币证明，而不是质押、消费或不可撤销承诺。

这意味着 like 可以作为生态参与信号，但不能单独作为强激励依据。未来评分或空投系统应把 like 当作一个低成本、可刷、需过滤的行为信号。

### 5.7 GovernanceVault 与 FeeManager

GovernanceVault 负责协议权限、角色和治理可执行动作的边界。FeeManager 负责费用等级和收费路径。

两者必须绑定官方 root，不能仅凭 registry id 或类型假设其可信。PaperProof 的安全边界依赖对象 ID 绑定，而不是只依赖 Move 类型。

### 5.8 GovernanceConfig 与 Proposal

GovernanceConfig 定义提案门槛、投票周期、总供应记录、当前 active proposal 和可执行治理动作。

Proposal 表示一次治理过程。它可以是 signal，也可以是 executable。治理执行不是单纯读取提案结果，而是在执行时重新校验提案、config、vault 和 action 状态。

这种重复校验避免了“投票通过时有效，执行时已失效”的治理空执行问题。

## 6. Artifact Type 模型

PaperProof 的 artifact type 是协议层概念，不是用户标签。

内置类型代表协议认为值得提供强语义支持的内容类别。例如：

- `preprint`：面向预印本、研究草稿和早期论文；
- `blog_post`：面向长文与公开说明；
- `technical_report`：面向技术报告、审计报告和工程文档；
- `dataset`：面向数据集；
- `software_release`：面向软件包、发布版本和源码包；
- `generic_file`：面向尚未细分但仍需纳入协议的普通文件。

这些类型不是永久封闭的。PaperProof 可以通过治理和合约升级增加新类型，但新增类型应满足高价值、可索引、可约束、可长期维护的标准。

类型治理的关键问题不是“用户能不能自由表达”，而是“协议核心是否应该承认该表达为一类可组合基础对象”。

## 7. Artifact Code

Artifact code 是面向用户、前端和链下引用的可读工件编号。当前设计采用类似：

`PaperProof-{type}-{epoch6}-{series_id_hex_12}`

它把类型、发布时期和 series id 短标识结合起来，避免全局递增计数器带来的并发瓶颈。

Artifact code 的意义不是替代 object id，而是提供更适合人类传播、文档引用和前端展示的协议级标识。真正的强身份仍来自 Sui object id 和官方 root 绑定。

## 8. Metadata Extensions

PaperProof 引入 metadata extensions，是为了在不频繁升级核心结构的情况下，为 series 和 version 增加有限扩展空间。

Metadata extensions 采用 key/value 形式，并受到数量、key 长度、value 长度和重复 key 限制。它们是展示和索引用途，不应成为核心权限、费用或治理安全逻辑的依据。

Series metadata 可以由 series owner 后续更新，并产生事件。Version metadata 一经提交不可修改。

这个区别反映了两种不同语义：

- series 是长期身份，可以维护展示信息；
- version 是历史承诺，应保持不可变。

## 9. Walrus 与 Sui 的分工

PaperProof 的分工原则可以概括为：Walrus 存内容，Sui 存承诺。

Walrus 适合承载原始文件和大体积 blob。Sui 适合承载经过强约束的小型状态、对象关系和事件。

因此，一个预印本 PDF 不应作为字节数组写进 Sui；但它的 hash、Walrus id、version id、series id、artifact code、owner 和发布事件必须在 Sui 上可验证。

这种分工带来三个优势：

- 成本可控：大内容不上链，减少对象膨胀；
- 验证清晰：链上 hash 与 Walrus 内容形成可验证关系；
- 应用灵活：前端和 indexer 可根据链上状态定位内容并构造丰富体验。

## 10. 评论与互动模型

PaperProof 的评论模型是协议内置互动层，而不是外部论坛插件。

每个 series 可以拥有一个官方 comments tree。评论树绑定 series，受 tree owner 管理。评论可以是链上短内容，也可以是 blob-backed 内容，后者适合更长文本或外部内容摘要。

评论状态机体现社区治理边界：

- active 表示正常展示；
- hidden 表示被管理方隐藏；
- deleted 表示删除或撤回语义；
- author 不能越过 tree owner 的隐藏决定；
- tree lock 可以阻止新增评论，但不需要冻结所有历史评论状态。

这种模型承认评论既是内容互动，也是治理对象。它既要支持表达，也要支持管理。

点赞模型与评论树分离。点赞使用 LikesBook，避免热门内容的点赞流量与评论流量争用同一个对象。当前点赞是 PPRF 持币证明，表达“该地址在点赞时拥有足够 PPRF”，不是质押、燃烧或强承诺。

## 11. PPRF 的协议作用

PPRF 是 PaperProof 生态的核心连接资产。

### 11.1 治理权

PPRF 用于治理提案、投票和协议参数调整。治理覆盖费用、角色、类型状态、升级相关权限和其他核心动作。

治理不是无限制执行。提案创建、投票、结算和执行都必须满足状态机规则。可执行治理动作还必须在执行时重新校验业务状态。

### 11.2 持币证明

PPRF 可以作为某些生态行为的持币证明。例如点赞要求用户持有足够 PPRF coin。这个机制建立了一种轻量参与门槛，但并不意味着用户锁定或消耗了 PPRF。

因此，持币证明适合表达“参与资格”或“弱信号”，不适合单独承载强激励。

### 11.3 费用与生态参与

PPRF 未来可以连接费用折扣、协议收入分配、贡献权重、生态身份、空投规则和社区治理权重。当前合约为这些路径保留了治理、事件和对象基础，但不急于把所有经济机制一次性固化。

这是 PaperProof 的重要设计选择：核心协议先提供可信记账和治理边界，再让激励系统在真实使用数据和社区共识中演进。

### 11.4 风险边界

PPRF 治理如果只基于持币，不锁仓或不使用快照，就天然存在借币、短期集中投票和女巫行为风险。

PaperProof 可以在合约外通过托管策略、治理流程、时间窗口、快照、前端提示和社区规则缓解这些问题。核心合约不必一次性解决所有社会层治理问题，但必须清楚暴露其安全边界。

## 12. 治理扩展路径

PaperProof 的扩展路径由治理与升级共同构成。

治理适合处理参数、状态和权限变化，例如：

- 启用或停用 artifact type；
- 调整费用等级；
- 修改治理阈值；
- 变更角色地址；
- 管理可执行 action；
- 表达社区 signal。

合约升级适合处理结构和能力变化，例如：

- 新增 artifact type 的结构化发布函数；
- 新增字段；
- 修改对象模型；
- 引入新的协议模块；
- 修复安全问题。

PaperProof 不追求完全不可升级的静态协议。相反，它承认早期协议需要演进，但要求升级路径受权限、治理和文档约束。

这种“治理 + 升级”的模式适合一个需要长期扩展类型和生态功能的内容协议。

## 13. 费用模型

PaperProof 的费用模型由 FeeManager 和 GovernanceVault 支撑。发布、追加版本、评论等行为可以根据协议配置收取费用。

费用模型的关键不是短期收入最大化，而是形成可治理的资源边界：

- 防止垃圾发布和垃圾评论；
- 为协议运营提供可调节参数；
- 支持未来 treasury 和生态激励；
- 在不同 artifact type 之间表达不同成本。

费用可以为 0，也可以通过治理调整。免费并不代表没有协议边界，因为对象身份、类型状态、长度约束和官方路径仍然受到合约约束。

## 14. 事件与 Indexer

PaperProof 的事件系统是协议与链下世界之间的主要接口。

Indexer 可以通过事件重建：

- series 发布；
- version 追加；
- artifact code；
- owner 变化；
- metadata 更新；
- comments tree 和 likes book 绑定；
- 评论与回复；
- 点赞与取消点赞；
- 治理提案、投票、结算和领取；
- 费用和角色变化。

但 indexer 必须遵守可信入口原则：

- 以官方 PaperProofRoot 为根；
- 校验 TypeRegistry、FeeManager、GovernanceVault、GovernanceConfig 的官方对象 ID；
- 只把官方 publishing 路径创建的 comments tree 和 likes book 当作协议内对象；
- 不把裸事件、假对象事件或旧部署事件混入当前协议统计；
- 对 like、comment、publish 等可刷行为建立反作弊解释层。

事件不是数据库替代品，而是可验证事实流。Indexer 的价值在于把这些事实组织成可查询、可展示、可评分的应用数据。

## 15. 并发与对象设计

Sui 的对象模型要求协议设计认真处理共享对象竞争。

PaperProof 在几个关键位置做了并发取舍：

- artifact code 不依赖全局递增计数器，避免首发发布集中卡在同一个对象；
- comments tree 和 likes book 分离，降低热门作品互动竞争；
- comment 和 like 不强制读取全局 root，以减少高频互动路径的共享对象依赖；
- pause 主要控制 publish 和 add version，而不冻结所有互动行为；
- 类型治理和发布路径保留较强对象绑定，因为它们是低频高价值操作。

这体现了一个原则：低频高价值路径可以更强约束，高频互动路径应尽量减少不必要共享对象依赖。

## 16. 安全边界

PaperProof 的安全边界建立在几个核心假设上。

第一，官方对象 ID 是可信根。只检查类型或 registry id 不够，关键路径必须校验对象来源。

第二，内容真实性不由合约自动背书。合约能证明某地址发布了某 hash，不能证明该内容原创、正确、无侵权或学术有效。

第三，事件需要过滤。任何基于事件的积分、空投或声誉系统都必须使用官方对象绑定和反滥用规则。

第四，PPRF 持币证明不是锁仓。基于持币证明的互动信号应被视为弱信号。

第五，升级权限是协议风险的一部分。PaperProof 选择可升级性以支持早期演进，因此必须通过治理、文档、公告和多方监督管理升级风险。

## 17. 协议愿景

PaperProof 的长期愿景是成为数字工件的可信发布与参与层。

它不试图替代所有内容平台，也不试图把所有社会判断写成合约。它要做的是提供一个足够强的协议底座，让数字工件拥有可验证身份、可追踪版本、可治理类型、可组合互动和可解释事件。

在这个底座上，预印本可以被引用，软件发布可以被验证，数据集可以被索引，评论可以被组织，点赞可以成为参与信号，治理可以调整协议方向，PPRF 可以把贡献、费用、参与和长期生态价值连接起来。

PaperProof 的核心不是“把内容放到链上”，而是“让内容拥有可验证的协议生命”。

## 18. 结语

PaperProof 的核心合约设计追求一种平衡：

- 内容不上链，但内容承诺上链；
- 类型不无限开放，但可治理扩展；
- metadata 可扩展，但受强约束；
- version 可追加，但不可篡改；
- 互动可低成本发生，但不被简单等同于强激励；
- 事件广泛开放，但必须通过官方对象过滤；
- PPRF 连接治理与生态，但其社会层风险需要持续治理。

这套设计使 PaperProof 更接近一个长期协议，而不是一次性应用。它为数字工件建立可验证结构，为社区参与留下演进空间，也为未来围绕内容、声誉、贡献和激励的生态系统提供了可依赖的链上基础。
